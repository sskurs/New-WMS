Vendor Returns Implementation Guide

This document details the transition from a siloed Customer RMA process to a unified Return management architecture that supports both Customer (B2C) and Vendor (B2B) returns.

1. Database Design

To maximize code reuse, we leverage a polymorphic approach or a unified header table with a discriminator.

Unified Return Header Table

We extend the existing Customer RMA schema to handle Vendor specific fields while sharing core properties like reasons, status, and audit trails.

CREATE TABLE ReturnMaster (
    id INT PRIMARY KEY AUTO_INCREMENT,
    returnType VARCHAR(20) NOT NULL, -- 'Customer' or 'Vendor'
    referenceId VARCHAR(20), -- OrderId for customers, PONumber for vendors
    entityId INT, -- CustomerID or SupplierID
    reason VARCHAR(255),
    status VARCHAR(50), -- Shared states: Pending, Approved, Shipped, Completed
    priority VARCHAR(20),
    totalValue DECIMAL(10,2), -- Refund for customers, Credit Note for vendors
    createdAt DATETIME DEFAULT CURRENT_TIMESTAMP,
    isArchived INT DEFAULT 0
);

CREATE TABLE ReturnDetails (
    id INT PRIMARY KEY AUTO_INCREMENT,
    returnId INT, -- FK to ReturnMaster
    productId INT,
    quantity INT,
    unitCost DECIMAL(10,2),
    FOREIGN KEY (returnId) REFERENCES ReturnMaster(id)
);

2. Code Reuse Strategy

The Strategy: Use a shared ReturnService that handles the state machine (status transitions) and inventory logic, while delegating entity-specific tasks (Financials) to sub-handlers.

Customer (RMA) Logic

  • Restocks inventory (Warehouse IN)
  • Triggers payment gateway refund
  • Links to original Sales Order
  • Affects Customer Loyalty points

Vendor Return Logic

  • Decrements inventory (Warehouse OUT)
  • Generates a Supplier Credit Note
  • Links to original Purchase Order
  • Affects Supplier Rating

Shared Controller Pattern

Instead of separate controllers, use a unified endpoint with a type parameter to reduce boilerplate.

// Example pseudo-code for a unified handler
async function handleReturnCreation(payload) {
    // 1. Shared Validation (Qty checks, valid ProductId)
    validateGenericReturnData(payload);

    // 2. Entity Specific Stock Logic
    if (payload.type === 'Vendor') {
        await inventoryService.decrementStock(payload.items);
    } else {
        await inventoryService.prepareForRestock(payload.items);
    }

    // 3. Shared Persistence
    return await db.ReturnMaster.create(payload);
}

3. Backend API Specification

Create Vendor Return

POST /api/warehouse-management/returns/vendor/create

Initiates a return to a supplier. This endpoint reuse the same validation middleware as the customer RMA endpoint.

Request Payload

{
    "supplierId": 105,
    "purchaseOrderId": "PO-2025-001",
    "reason": "Damaged on arrival",
    "status": "Pending",
    "items": [
        { "productId": 12, "quantity": 5, "unitCost": 450.00 }
    ],
    "notes": "Batch #882 expired"
}

Get Returns List

GET /api/warehouse-management/returns/{type}/all

A single endpoint to fetch either Customer or Vendor returns based on the URL parameter.

Update Return Status

POST /api/warehouse-management/returns/update-status

Used to move returns through the lifecycle. For Vendor Returns, moving to "Shipped" automatically triggers inventory deduction.

Request Payload

{
    "returnId": 502,
    "status": "Shipped"
}

4. Summary of Reuse

  • Validation: Shared Zod/Joi schema for checking product lists and quantities.
  • UI Components: Use the same "Add Item" table component for both RMA and Vendor Return wizards.
  • Inventory Service: A single StockMovement table records all changes, whether it's a customer restocking or a vendor return decrementing.
  • Reporting: Unified valuation reports can pull from ReturnMaster filtering by returnType.
