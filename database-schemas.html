WMS Database Schemas

This document details the proposed database table structures for key warehouse operations: Picking, Packing & Shipping, and Cycle Counting. These schemas are designed for clarity, performance, and referential integrity.

Picking Operations

The picking process begins when an order is ready to be fulfilled. We need a PickLists table to represent the overall task and a PickListItems table for the specific items to be gathered from warehouse locations.

1. `PickLists` Table

This table acts as the header for a picking task, linking it to a customer order and assigning it to a specific user (the picker).

CREATE TABLE PickLists (
    PickListID INT PRIMARY KEY AUTO_INCREMENT,
    OrderID INT NOT NULL, -- Foreign Key to the Orders table
    AssignedToUserID INT, -- Foreign Key to the Users table (the picker)
    Status VARCHAR(50) NOT NULL DEFAULT 'Pending', -- e.g., 'Pending', 'In Progress', 'Picked'
    CreatedAt DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    CompletedAt DATETIME,

    FOREIGN KEY (OrderID) REFERENCES Orders(OrderID),
    FOREIGN KEY (AssignedToUserID) REFERENCES Users(UserID),
    INDEX idx_picklists_status (Status) -- Index for quick filtering by status
);

2. `PickListItems` Table

This table details each specific item that needs to be picked for a given PickList, including the source location and quantities.

CREATE TABLE PickListItems (
    PickListItemID INT PRIMARY KEY AUTO_INCREMENT,
    PickListID INT NOT NULL, -- Foreign Key to the PickLists table
    ProductID INT NOT NULL, -- Foreign Key to the Products table
    LocationID INT NOT NULL, -- Foreign Key to the Locations table (where to pick from)
    QuantityToPick INT NOT NULL,
    QuantityPicked INT NOT NULL DEFAULT 0,
    IsPicked BOOLEAN NOT NULL DEFAULT FALSE,

    FOREIGN KEY (PickListID) REFERENCES PickLists(PickListID),
    FOREIGN KEY (ProductID) REFERENCES Products(ProductID),
    FOREIGN KEY (LocationID) REFERENCES Locations(LocationID)
);

Packing & Shipping Operations

Once all items on a pick list are marked as 'Picked', the order moves to the packing and shipping stage. This involves creating a physical package and preparing it for shipment.

3. `PackedOrders` Table

This table represents a physically packed box or shipment, ready for labeling and final dispatch. It's the bridge between the picking process and the final shipment to the customer.

CREATE TABLE PackedOrders (
    PackedOrderID INT PRIMARY KEY AUTO_INCREMENT,
    OrderID INT NOT NULL UNIQUE, -- A direct, unique link to the order
    PackedByUserID INT, -- Foreign Key to the Users table
    PackedAt DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    ShippingLabelURL VARCHAR(255), -- URL to the generated shipping label
    TrackingNumber VARCHAR(100),
    Status VARCHAR(50) NOT NULL DEFAULT 'Packed', -- e.g., 'Packed', 'Shipped'

    FOREIGN KEY (OrderID) REFERENCES Orders(OrderID),
    FOREIGN KEY (PackedByUserID) REFERENCES Users(UserID),
    INDEX idx_packedorders_status (Status)
);

Cycle Counting Operations

Cycle counting is a critical inventory auditing process for maintaining accuracy. It involves periodically counting a small subset of inventory rather than performing a full physical count.

4. `CycleCounts` Table

This table stores the header information for a cycle count task, such as its status, who initiated it, and when it was created and completed.

CREATE TABLE CycleCounts (
    CycleCountID INT PRIMARY KEY AUTO_INCREMENT,
    Status VARCHAR(50) NOT NULL DEFAULT 'Pending', -- e.g., 'Pending', 'In Progress', 'Completed', 'Adjusted'
    CreatedAt DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    CreatedByUserID INT, -- Foreign Key to the Users table
    CompletedAt DATETIME,

    FOREIGN KEY (CreatedByUserID) REFERENCES Users(UserID),
    INDEX idx_cyclecounts_status (Status)
);

5. `CycleCountItems` Table

This table holds the specific items and locations that are part of a cycle count task. It records the state of inventory at the time of the count, the actual counted quantity, and the resulting variance.

CREATE TABLE CycleCountItems (
    CycleCountItemID INT PRIMARY KEY AUTO_INCREMENT,
    CycleCountID INT NOT NULL, -- Foreign Key to the CycleCounts table
    ProductID INT NOT NULL,
    LocationID INT NOT NULL,
    SystemQuantity INT NOT NULL, -- The quantity recorded in the system when the count was created
    CountedQuantity INT, -- The actual quantity counted by the user
    CountedAt DATETIME,
    CountedByUserID INT, -- Foreign Key to the Users table
    Variance INT GENERATED ALWAYS AS (CountedQuantity - SystemQuantity) STORED, -- Calculated difference

    FOREIGN KEY (CycleCountID) REFERENCES CycleCounts(CycleCountID),
    FOREIGN KEY (ProductID) REFERENCES Products(ProductID),
    FOREIGN KEY (LocationID) REFERENCES Locations(LocationID),
    FOREIGN KEY (CountedByUserID) REFERENCES Users(UserID),
    UNIQUE KEY uq_cycle_item (CycleCountID, ProductID, LocationID) -- Prevents duplicate items in the same count
);
