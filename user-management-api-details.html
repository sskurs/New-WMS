User Management API Details

This document provides the complete API specification and database schema for the User Management module, including all relevant fields as used by the frontend application.

Database Table Schema

The following SQL schema represents the `Users` table structure, inferred from the API payloads and application's data types.

CREATE TABLE Users (
    UserID INT PRIMARY KEY AUTO_INCREMENT,
    UserName VARCHAR(255) NOT NULL UNIQUE,
    PasswordHash VARCHAR(512) NOT NULL,
    Salutation VARCHAR(10),
    FullName VARCHAR(255),
    DisplayName VARCHAR(255),
    DOJ DATE, -- Date of Joining
    Gender VARCHAR(10),
    RoleID INT NOT NULL, -- Foreign Key to a Roles table
    BranchID INT,
    DepartmentID INT,
    OfficialEmail VARCHAR(255) NOT NULL UNIQUE,
    PersonalEmail VARCHAR(255),
    ContactNo VARCHAR(20),
    Remarks TEXT,
    IsActive BIT NOT NULL DEFAULT 1, -- 1 for Active, 0 for Inactive
    IsArchived BIT NOT NULL DEFAULT 0,
    CreatedOn DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    CreatedBy INT,
    ModifyOn DATETIME,
    ModifyBy INT,

    FOREIGN KEY (RoleID) REFERENCES Roles(RoleID)
);

API Endpoints

GET /User/GetAll

Get User by ID

GET /User/GetById?id=1

Retrieves the details for a single user by their unique ID.

Success Response Payload (200 OK)

// The response is a single user object, identical in structure to an object in the 'data' array from the GetAll endpoint.
{
    "userId": 1,
    "userName": "admin@propixtech.com",
    // ... all other user fields as per the GetAll response structure
}

Create User

POST /User/Create

Creates a new user account. This payload includes all fields required by the frontend form.

Request Payload

{
    "userName": "new.picker",
    "password": "AStrongPassword123!", // Note: Frontend sends 'password', backend expects 'passwordHash'
    "salutation": "Mr.",
    "fullName": "Ravi Kumar",
    "displayName": "Ravi K.",
    "doj": "2025-10-01",
    "gender": "Male",
    "roleId": 2, // 2 for 'Picker'
    "branchId": 1,
    "departmentId": 3,
    "emailId": "ravi.k@example.com", // Maps to 'officialEmail'
    "personalEmail": "ravi.personal@email.com",
    "contactNo": "9876543210",
    "remarks": "New hire for the picking team."
}

Update User

PUT /User/Update

Updates an existing user's details. The `userId` is mandatory. `passwordHash` can be an empty string if not updating the password.

Request Payload

{
    "userId": 5,
    "userName": "updated.user",
    "passwordHash": "", // Empty string if not changing password
    "salutation": "Ms.",
    "fullName": "Priya Sharma",
    "displayName": "Priya S.",
    "doj": "2025-08-15",
    "gender": "Female",
    "roleId": 4, // 4 for 'Warehouse Manager'
    "branchId": 1,
    "departmentId": 2,
    "officialEmail": "priya.s@example.com",
    "personalEmail": "",
    "contactNo": "9123456780",
    "remarks": "Promoted to Warehouse Manager."
}

Update User Status

PUT /User/UpdateStatus

A specific endpoint to activate or deactivate a user account.

Request Payload

{
    "userId": 5,
    "isActive": 0 // 0 to Deactivate, 1 to Activate
}

Delete User

DELETE /User/Delete?userId=99

Deletes a user by their ID, passed as a query parameter. No request body is needed.
